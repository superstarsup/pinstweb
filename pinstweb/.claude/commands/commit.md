# /commit — 이슈 번호 필수 커밋

이 저장소의 모든 커밋 메시지에는 **관련 GitHub 이슈 번호(`#N`)가 반드시 포함**되어야 한다. 이 스킬은 그 규칙을 강제한다.

## 실행 절차

1. **변경 내용 확인** — 다음을 **병렬**로 실행:
   - `git status`
   - `git diff --staged`
   - `git diff`
   - `git log --oneline -10`
   - `git branch --show-current`

2. **이슈 번호 확정** — 다음 우선순위로 결정:
   1. 사용자가 `/commit #12` 처럼 인자로 명시한 경우 → 그 번호 사용
   2. 브랜치 이름에서 추출 가능(`feature/12-foo`, `issue-12`, `fix/12-...`) → 그 번호 사용
   3. 위 둘 다 없으면 `gh issue list --state open --limit 20` 을 실행해 목록을 보여주고 사용자에게 어떤 이슈와 연결되는지 묻는다. **추측하지 말 것.**
   4. 관련 이슈가 정말로 없다면 먼저 이슈를 만들 것을 제안한다 (`gh issue create`). 이슈 없이 커밋하지 않는다.

3. **이슈 유효성 확인** — 결정된 번호를 `gh issue view <N>` 으로 확인해 실제 존재하며 닫히지 않았는지(또는 닫힌 이슈를 재작업 중임을 사용자가 인지하는지) 검증한다.

4. **커밋 메시지 작성** — 아래 형식을 따른다:
   ```
   <타입>: <한 줄 요약> (#<이슈번호>)

   <본문: 왜 이 변경이 필요한지, 무엇이 바뀌었는지>

   Refs #<이슈번호>
   ```
   - 타입 예시: `fix`, `feat`, `refactor`, `docs`, `chore`, `style`
   - 이슈가 이 커밋으로 완전히 해결되면 `Refs` 대신 `Closes #<번호>` 사용
   - 요약은 명령형 한국어 또는 영어, 50자 이내 권장
   - 제목과 본문 둘 중 최소 한 곳에는 `#N` 이 반드시 들어가야 한다

5. **스테이징 및 커밋** — 사용자가 특정 파일만 지정하지 않았다면 현재 스테이지된 파일을 그대로 사용. 스테이지가 비어 있으면 어떤 파일을 커밋할지 먼저 확인한다. `git add -A` / `git add .` 은 **사용하지 않는다** (민감 파일 실수 방지).

   HEREDOC 으로 커밋:
   ```bash
   git commit -m "$(cat <<'EOF'
   fix: banner 테스트 문자열 제거 (#2)

   리뷰에서 지적된 `sssss`, `ㄴㄴㄴㄴㄴ11` 잔존 문자열을 정리.

   Closes #2

   Co-Authored-By: Claude Opus 4.7 <noreply@anthropic.com>
   EOF
   )"
   ```

6. **검증** — `git log -1` 로 메시지에 `#N` 이 들어갔는지 최종 확인 후 `git status` 로 결과를 사용자에게 보여준다.

## 규칙 (엄수)

- **이슈 번호가 없는 커밋은 만들지 않는다.** 사용자가 "그냥 커밋해" 라고 해도 해당 이슈가 무엇인지 반드시 확인한다.
- `git commit --amend` 는 사용자가 명시적으로 요청하지 않는 한 사용하지 않는다. 새 커밋을 만든다.
- `--no-verify` / `--no-gpg-sign` 금지.
- 푸시는 사용자가 별도로 요청할 때만 수행한다.
- pre-commit hook 이 실패하면 원인을 수정하고 **새 커밋**을 만든다. amend 하지 않는다.

## 인자

- `$ARGUMENTS` — 비어 있거나 `#12` / `12` 같은 이슈 번호가 올 수 있다. 있으면 위 절차 2단계의 1순위로 사용.
